## 2. 專案背景與目標

本系統旨在建立公司內部之內控治理入口網站，將合規公告、制度資源、控制點管理、缺失追蹤、證據留存與稽核軌跡整合為單一平台，以支援治理、風險與法遵管理需求。

系統建置目標如下：

1. 建立可追蹤、可驗證、可稽核之內控治理流程。
2. 形成 `風險 -> 控制點 -> 測試 -> 缺失 -> 修復 -> 驗證 -> 結案` 之閉環管理。
3. 降低制度傳達落差、證據散落、缺失追蹤不完整及權限控管不足之風險。
4. 提供管理層即時掌握控制有效性、缺失處理進度及整體治理成熟度之資訊。

## 6. 業務流程總覽

### 6.1 合規公告流程

`草稿 -> 審核中 -> 已發布 -> 已歸檔`

### 6.2 控制測試流程

`排程建立 -> 待執行 -> 測試中 -> 待覆核 -> 已完成`

若測試不通過，系統須可自動或手動建立對應 Issue。

### 6.3 Issue 處理流程

`Open -> Assigned -> Plan Submitted -> In Progress -> Pending Verification -> Closed`

如需延期，應進入：

`延期申請中 -> 已核准延期 / 已駁回延期`

### 6.4 例外治理流程

高風險事項逾期、控制測試逾期、重大公告未讀者，系統須依規則自動催辦與升級通知。

## 8. 非功能需求

### 8.1 安全性

1. 系統應整合公司 SSO 或 AD。
2. 離職或停權帳號應立即失效。
3. 所有權限判斷應於後端再次驗證。
4. 關鍵資料異動應寫入不可任意刪改之 Audit Trail。
5. 敏感資料匯出應支援欄位遮罩。

### 8.2 稽核軌跡

系統至少應記錄以下操作：

1. 登入、登出
2. 查詢敏感頁面
3. 新增、修改、刪除、停用
4. 狀態變更
5. 匯出報表
6. 檔案下載

Audit Trail 應至少包含：

`時間、使用者、角色、模組、動作、物件編號、異動前值、異動後值、來源 IP`

### 8.3 資料保留

1. Issue、控制測試、證據、閱讀紀錄及操作日誌應保留至少 5 年。
2. 系統原則上不提供實體刪除，採邏輯刪除或封存。
3. 封存資料應可供稽核查閱。

### 8.4 效能

1. 一般查詢頁面於正常負載下應於 3 秒內回應。
2. 清單頁應支援分頁、排序、條件篩選。
3. 報表匯出應支援背景處理與完成通知。

---

## 9. 介面與易用性需求

1. 首頁應提供三大區塊：最新公告、我的待辦事項、常用制度資源。
2. 所有清單頁應支援搜尋、篩選、排序及匯出。
3. 高風險狀態應以紅色明確標示，中風險以橘色標示。
4. 逾期事項應有視覺警示與待辦計數。
5. 畫面文字與欄位名稱應採一致術語，與本規格書資料字典一致。

## 13. 需求摘要

本規格書將原始的 `News`、`Hyperlink`、`Issue Tracker` 需求，正式轉化為下列四個核心模組：

1. 合規公告中心：強調制度傳達與閱讀證明。
2. 制度資源庫：強調制度存取正確性與版本管理。
3. 缺陷修復與追蹤系統：強調缺失閉環、SLA、驗證與結案。
4. 控制點矩陣管理：強調控制設計、測試、覆蓋率與風險治理。

本版本已可作為後續系統分析、資料表設計、畫面設計、API 設計與測試案例撰寫之基礎文件。
